# Part 1: 项目代码核心知识

## 第 1 层：tools.py — 工具系统

### 核心问题：大模型怎么"用工具"？

大模型只会输出文字，它**不执行工具**。它只是"说"它想执行什么。真正执行的是 Python 代码。

流程：

```
你的代码 → 告诉大模型："你有这些工具可以用，每个工具长这样"
大模型   → 返回："我想调用 read_file 工具，参数是 path='test.txt'"
你的代码 → 看到这个意图，真正去读文件，拿到结果
你的代码 → 把结果再发给大模型
大模型   → 根据结果继续回答
```

这个过程叫 **Tool Call**（也叫 Function Calling）。

### 工具的元数据：JSON Schema

告诉大模型每个工具的：
1. **名字** — 叫什么
2. **描述** — 干什么用的（大模型靠这个决定用不用它）
3. **参数结构** — 用 JSON Schema 描述

```json
{
  "type": "object",
  "properties": {
    "path": {
      "type": "string",
      "description": "要读取的文件路径"
    }
  },
  "required": ["path"]
}
```

翻译：这个工具接受一个对象，里面有一个叫 `path` 的字符串参数，必填。

### 关键代码结构

**BaseTool**：所有工具的基类，定义统一接口

```python
class BaseTool:
    def __init__(self, name, description, parameters):
        self.name = name
        self.description = description
        self.parameters = parameters

    def to_openai_function(self):     # 翻译成大模型能看懂的格式
        return {...}

    async def execute(self, **kwargs):  # 子类必须实现
        raise NotImplementedError
```

`raise NotImplementedError` 的作用：**父类定规矩，子类来实现。** 强制约定，防止写出残缺工具。

**ToolRegistry**：工具注册中心

```python
class ToolRegistry:
    def __init__(self):
        self.tools = {}
        self.register(ReadFileTool())
        self.register(WriteFileTool())
        ...

    def get_definitions(self):  # 返给大模型看的工具列表
        return [tool.to_openai_function() for tool in self.tools.values()]

    async def execute(self, name, arguments_json):
        tool = self.tools[name]
        kwargs = json.loads(arguments_json)
        return await tool.execute(**kwargs)
```

### 数据流

```
大模型返回：{"name": "read_file", "arguments": '{"path": "test.txt"}'}
    ↓
ToolRegistry.execute("read_file", '{"path": "test.txt"}')
    ↓
json.loads → {"path": "test.txt"}
    ↓
ReadFileTool.execute(path="test.txt")
    ↓
返回文件内容字符串
```

### 关键概念：async/await

加了 `async` 的函数叫**异步函数**，调用时要加 `await`。
好处：读文件、执行命令这些慢操作不会阻塞程序。

## 第 2 层：memory.py — 记忆系统

### 核心问题：大模型为什么需要记忆？

大模型本身**无状态**。每次调用 API，它都不记得上一次说了什么。

多轮对话的本质：**每次都把历史对话全部重新发给它。**

```
第 1 轮发送：[用户: "你好"]
第 2 轮发送：[用户: "你好", 助手: "你好！", 用户: "我叫小明"]
第 3 轮发送：[..., 助手: "你好小明", 用户: "我叫什么？"]
```

### 两种记忆

| 类型 | 存在哪 | 是什么 |
|---|---|---|
| 短期记忆 | `default_history.json` | 最近 N 轮对话，每次请求都带上 |
| 长期记忆 | `MEMORY.md` | 重要事实，写进 System Prompt |

短期记忆**自动管理**，长期记忆**Agent 主动写入**（通过 `write_file` 工具）。

### 重点：安全截断

```python
def get_messages(self, window_size=20):
    if len(self.messages) <= window_size:
        return self.messages

    start_idx = max(0, len(self.messages) - window_size)

    # 关键：找到最近的 user 消息作为起点
    while start_idx > 0 and self.messages[start_idx].get("role") != "user":
        start_idx -= 1

    return self.messages[start_idx:]
```

**为什么不能简单截断？**

如果截断后第一条是 `tool` 消息，但对应的 `assistant tool_calls` 被截掉了，会变成"孤儿 tool 结果"——大模型不知道这个结果对应哪个调用，API 直接报错。

所以必须从 `user` 消息开始截断，保证工具调用链完整。

## 第 3 层：skills.py — 技能插件系统

### 核心问题：怎么让 Agent 具备专业能力，又不浪费 token？

如果把所有技能指南都塞进 System Prompt，简单问题也会浪费大量 token。

### 解法：两档加载策略

```
always_load: true  → 全文注入 System Prompt（核心技能，每次都用）
always_load: false → 只告诉大模型"有这个技能，路径在这里"
                     大模型需要时自己调用 read_file 去读
```

### SKILL.md 结构

```markdown
---
name: data-analysis
description: 数据分析技能
always_load: false
---

# 数据分析技能

当用户需要分析数据时，按以下步骤操作……
```

`---` 包裹的部分是 **YAML frontmatter**（元数据），下面是正文。

### 解析逻辑

```python
def _parse_markdown_with_frontmatter(self, filepath):
    text = open(filepath).read()
    match = re.match(r'^---\s*\n(.*?)\n---\s*\n(.*)', text, re.DOTALL)
    if match:
        meta = yaml.safe_load(match.group(1))
        content = match.group(2).strip()
        return meta, content
    return {}, text.strip()
```

### 两个对外方法

- `get_always_skills_prompt()` — 返回 always_load 技能的完整内容
- `build_skills_summary_prompt()` — 返回非 always_load 技能的名称+描述+路径（不含正文）

### 关键理解

大模型**一开始就知道有哪些技能**（通过摘要）。需要用时主动调 `read_file` 去读完整内容。这是"主动查手册"模式。

## 第 4 层：context.py — 上下文组装

### messages 数组结构

发给大模型的不是文字，而是消息数组：

| role | 含义 |
|---|---|
| `system` | 系统设定，第一条，只有一条 |
| `user` | 用户说的话 |
| `assistant` | 大模型回复 |
| `tool` | 工具执行结果 |

```python
[
    {"role": "system",    "content": "你是 tinybot..."},
    {"role": "user",      "content": "帮我读 test.txt"},      # 历史
    {"role": "assistant", "content": None, "tool_calls": [...]},
    {"role": "tool",      "content": "文件内容是 xxx"},
    {"role": "assistant", "content": "文件里写的是..."},
    {"role": "user",      "content": "谢谢"},                 # 当前
]
```

### build_messages() 组装顺序

```python
def build_messages(self, current_user_message):
    messages = []
    # 1. System Prompt
    messages.append({"role": "system", "content": self.build_system_prompt()})
    # 2. 历史对话（安全截断）
    messages.extend(self.memory.get_messages(window_size=20))
    # 3. 当前用户输入
    messages.append({"role": "user", "content": current_user_message})
    return messages
```

固定顺序：**系统设定 → 历史 → 当前输入**

### System Prompt 内部组装顺序

```
身份设定（时间、环境、工作区路径）
    +
常驻技能全文（always_load: true）
    +
长期记忆（MEMORY.md）
    +
可选技能摘要（always_load: false 的名称+路径）
```

最重要的放最前面，大模型对靠前内容注意力更强。

### 易错点：先 build_messages 再存 memory

```python
# 正确顺序
messages_payload = self.context.build_messages(user_message)  # 先用
self.memory.add_message({"role": "user", "content": user_message})  # 再存

# 反过来会导致用户消息重复（历史里有一份 + 当前又加一份）
```

模式总结：**先基于当前状态构建输出，再更新状态。**

## 第 5 层：loop.py — 事件循环

### ReAct 循环

```
普通聊天：用户问 → 模型答 → 结束

Agent：用户问 → 模型想 → 模型决定用工具 → 执行工具
       → 把结果给模型 → 模型继续想 → 可能再用工具 → ... → 最终回答
```

"想 → 行动 → 观察 → 再想"叫 **ReAct 模式**（Reasoning + Acting）。

### 核心代码（伪代码版）

```python
async def run(self, messages):
    current_messages = list(messages)
    max_iterations = 10  # 保险丝

    while iteration < max_iterations:
        # 1. 调用大模型
        response = await self.client.chat.completions.create(
            messages=current_messages, stream=True
        )

        # 2. 流式读取
        for chunk in response:
            if chunk 是文字:
                yield {"type": "text_delta", "content": "一个字"}
            if chunk 是工具调用:
                收集到 buffer 里

        # 3. 没有工具调用就退出
        if not tool_calls:
            break

        # 4. 顺序执行所有工具调用
        for tool_call in tool_calls:
            result = await self.tool_registry.execute(...)
            current_messages.append(tool结果)

        # 回到 while 顶部
```

### 关键点

**1. 退出条件：大模型不再要求调用工具**
大模型自己决定循环何时结束。

**2. 同一轮内多个工具是顺序执行**
不是每个工具调用都重新走一遍 while 循环。一轮里 for 循环执行完所有工具，再统一交给大模型。

```
while 第 1 轮：
  调用大模型 → 返回 2 个 tool_calls
  for 循环：执行 tool_1 → 执行 tool_2
  回到 while 顶部
while 第 2 轮：
  调用大模型（带两个工具结果）→ 返回纯文本
  break
```

**3. 流式 Tool Call 的碎片拼接**

流式返回时，工具调用是碎片化的：

```
chunk 1: name = "read"
chunk 2: name = "_file"
chunk 3: arguments = '{"pa'
chunk 4: arguments = 'th":'
chunk 5: arguments = '"test.txt"}'
```

用 `tool_call_buffer` 字典按 `index` 拼接：

```python
tool_call_buffer = {}
for tc in delta.tool_calls:
    idx = tc.index
    if idx not in tool_call_buffer:
        tool_call_buffer[idx] = {初始化}
    else:
        tool_call_buffer[idx]["function"]["arguments"] += tc.function.arguments
```

**4. max_iterations = 10 的意义**
防止大模型陷入死循环（一直调工具但永远不满意）。**安全兜底**，面试常考。

**5. yield 与 AsyncGenerator**

```python
yield {"type": "turn_end", "new_messages": current_messages[len(messages):]}
```

`current_messages[len(messages):]` 切掉旧的，只返回新产生的消息。给上层 agent.py 用来存入 memory。

## 第 7 层：app.py — HTTP 接口与 SSE

### SSE（Server-Sent Events）

普通 HTTP：发请求 → 处理 → 一次性返回 → 关闭
SSE：发请求 → 边处理边推 → 推完关闭

格式固定：

```
data: {"type": "text_delta", "content": "你"}\n\n
data: {"type": "text_delta", "content": "好"}\n\n
```

以 `data: ` 开头，以 `\n\n` 结尾。浏览器 `EventSource` API 自动解析。

### 核心接口

```python
@app.post("/api/chat")
async def chat_endpoint(req: ChatRequest):
    async def sse_generator():
        async for event in agent.chat_stream(req.message):
            data_str = json.dumps(event, ensure_ascii=False)
            yield f"data: {data_str}\n\n"

    return StreamingResponse(sse_generator(), media_type="text/event-stream")
```

`StreamingResponse` 不会等生成器跑完，每 yield 一次就发送一次。

### 安全细节：路径穿越防护

```python
if ".." in filename or "/" in filename or "\\" in filename:
    return {"status": "error", "message": "Invalid filename"}
```

防止攻击者用 `../../etc/passwd` 访问系统文件。

## 完整数据流（必背）

从用户输入"帮我读一下 test.txt"到看到回答：

```
1. app.py 收到 POST /api/chat
       ↓
2. 调用 agent.chat_stream("帮我读一下 test.txt")
       ↓
3. context.build_messages():
   - 组装 System Prompt（身份 + 常驻技能 + 长期记忆 + 可选技能摘要）
   - 从 memory.get_messages() 取历史（最近 20 条，安全截断）
   - 追加当前用户消息
       ↓
4. memory 存入用户消息
       ↓
5. loop.run(messages) 进入 ReAct 循环
       ↓
6. 第 1 轮：调用大模型 API（带 tools 定义）
   - 大模型返回 tool_call: {name: "read_file", arguments: '{"path": "test.txt"}'}
       ↓
7. tool_registry.execute("read_file", ...)
   - 找到 ReadFileTool，调用 execute(path="test.txt")
   - 返回文件内容
       ↓
8. 工具结果追加到 messages，回到 while 顶部
       ↓
9. 第 2 轮：再次调用大模型（带工具结果）
   - 模型组织语言，返回纯文本，无 tool_calls
   - break
       ↓
10. yield turn_end，携带 new_messages
       ↓
11. agent 收到 turn_end，把 new_messages 存入 memory
       ↓
12. 全程 text_delta 通过 SSE 实时推送给前端
```

**关键点：**
- 大模型**不执行工具**，只输出意图
- 这次循环了 **2 轮**（第 1 轮拿到工具调用，第 2 轮拿到最终回答）
- 每一步都是流式的

## 原创功能：RAG 知识库工具

### 功能描述

在 SmartFlow 里新增 `search_knowledge` 工具，Agent 需要查私域知识时自主调用，实现"先检索再回答"，减少幻觉。

### 技术栈

| 组件 | 选型 | 理由 |
|---|---|---|
| Embedding 模型 | `sentence-transformers` paraphrase-multilingual-MiniLM-L12-v2 | 免费本地运行，支持中英文，体积 ~120MB |
| 向量数据库 | `ChromaDB` | 纯 Python，零配置，持久化存储 |
| 文档切片 | 自实现（段落优先 + 固定长度兜底） | 不引入额外依赖 |

### 核心流程

```
离线建库：
  文档(.txt/.md) → 切片(chunk_size=300, overlap=50) → 向量化 → 存入 ChromaDB

在线检索：
  用户提问 → Agent 调用 search_knowledge(query) → 问题向量化
           → ChromaDB 余弦相似度检索 Top-K → 返回片段+来源+相似度分数
           → Agent 基于检索结果回答
```

### 改动文件

| 文件 | 改动 |
|---|---|
| `core/knowledge.py` | 新增，KnowledgeBase 类（切片/向量化/检索/持久化） |
| `core/tools.py` | 新增 SearchKnowledgeTool；ToolRegistry 接收 knowledge_base 参数 |
| `core/agent.py` | 初始化 KnowledgeBase，传给 ToolRegistry |
| `app.py` | 新增 /api/knowledge/add、/stats、/clear 三个接口 |
| `test_rag.py` | 新增，6 个测试全部通过（空库/导入/检索/不相关/文件/工具层） |

### 测试结果

```
相关问题（"RAG 有什么优势"）  → 相似度 0.76 / 0.65  ✓
不相关问题（"今天天气怎么样"）→ 相似度 0.51         ✓（明显低于相关内容）
```

### 面试讲法（30 秒版本）

> "我在 SmartFlow 里实现了 RAG 知识库工具。用 sentence-transformers 做 Embedding，ChromaDB 做向量存储。文档导入时先切片（段落优先，固定长度兜底，带重叠防止语义截断），再批量向量化存入 ChromaDB。检索时把用户问题向量化，用余弦相似度找 Top-K 最相关片段，作为工具结果返回给 Agent。整个过程本地运行，不消耗额外 API。我写了 6 个测试验证了空库、导入、检索、不相关问题等场景。"

### 面试追问准备

**Q：为什么切片要有重叠（overlap）？**
> 防止一句完整的话被切断在两个 chunk 的边界。比如"RAG 的优势是减少幻觉，知识可更新"如果被切成"RAG 的优势是减少幻觉"和"知识可更新"，检索"知识更新"时可能找不到完整语义。重叠 50 字保证边界处的语义连续。

**Q：余弦相似度和欧氏距离有什么区别？为什么用余弦？**
> 欧氏距离受向量长度影响，长文本的向量模长大，会导致距离偏大。余弦相似度只看方向不看长度，语义相近的文本余弦值接近 1，与文本长短无关。文本检索场景用余弦更准确。

**Q：ChromaDB 和 FAISS 有什么区别？**
> FAISS 是 Meta 开源的纯向量检索库，性能极高但只做检索，不存原始文本，需要自己管理 ID 和文档的对应关系。ChromaDB 是完整的向量数据库，自带持久化、元数据存储、文档管理，开发更方便。百万级以下数据量 ChromaDB 够用，超大规模才需要 FAISS 或 Milvus。

**Q：RAG 有什么局限性？**
> 三个主要问题：1）检索质量依赖切片策略，切得太碎或太长都影响召回；2）Embedding 模型对专业领域词汇（医疗、法律）效果差，需要领域微调；3）检索到的内容不一定是最有用的，可能需要 Reranker 二次排序提升精度。

## 原创功能：HITL 人工审批机制

> 这是你在 TinyAgent 基础上独立开发的原创功能，面试时的核心亮点。

### 功能描述

当 Agent 想执行高风险工具（如 Shell 命令）时，**先暂停**，向前端推送审批请求，等用户点击同意/拒绝后再继续。

### 改动文件清单

| 文件 | 改动内容 |
|---|---|
| `core/tools.py` | BaseTool 加 risk_level 字段；新增 ApprovalManager 类；ToolRegistry 加 get_risk_level() |
| `core/loop.py` | 执行工具前检查 risk_level，high 级触发审批流程 |
| `core/agent.py` | 创建 ApprovalManager 实例传给 loop；暴露 resolve_approval() 方法 |
| `app.py` | 新增 POST /api/approve 接口 |
| `static/index.html` | 监听 approval_required 事件，显示弹窗；提交审批结果 |

### 风险分级

| 等级 | 工具 | 处理方式 |
|---|---|---|
| `low` | read_file | 直接执行 |
| `medium` | write_file、edit_file | 直接执行 |
| `high` | exec（Shell 命令） | 执行前需用户审批 |

### 核心机制：asyncio.Event 异步通信

```
loop.py（等待方）                    app.py（响应方）
    ↓                                    ↓
create_request() → approval_id          收到 POST /api/approve
    ↓                                    ↓
yield approval_required 事件 → SSE → 前端弹窗
    ↓                                    ↓
await event.wait() ← 挂起              用户点击 → resolve(id, True)
    ↓                                    ↓
event.set() 被触发 ← 唤醒 ←────────────┘
    ↓
approved=True → 执行工具
```

**为什么需要 ApprovalManager？**
loop.py 和 app.py 是两个独立协程，无法直接共享变量。ApprovalManager 是它们之间的**共享桥梁**，用 `approval_id` 作为唯一钥匙精确匹配请求和响应。

**为什么用 asyncio.Event 而不是 sleep 轮询？**
`await event.wait()` 挂起时会**让出 CPU**，服务器可以继续处理其他用户的请求。sleep 轮询会持续占用 CPU，浪费资源。

**为什么 finally 块要清理 pending？**
防止**内存泄漏**。每次审批完成（无论同意/拒绝/超时）都必须从字典中删除，否则字典无限增长最终耗尽内存。

**为什么拒绝时也要往 messages 里塞 tool 消息？**
OpenAI 协议要求每个 tool_call 必须有对应的 tool 响应。如果不塞，会产生"孤儿 tool_call"，下一轮 API 调用直接报错。

### 面试讲法（30 秒版本）

> "我在 TinyAgent 基础上实现了 Human-in-the-Loop 人工审批机制。核心思路是：在 AgentLoop 执行高风险工具前，先 yield 一个 approval_required 事件通过 SSE 推给前端弹窗，然后用 asyncio.Event 挂起当前协程等待用户响应。用户点击后，前端 POST /api/approve，后端调用 event.set() 唤醒挂起的协程，根据结果决定执行还是跳过工具。整个过程异步非阻塞，不影响其他用户的请求。我还写了 mock 测试验证了同意和拒绝两条路径。"

### 面试追问准备

**Q：为什么不直接在 loop.py 里 await 一个 HTTP 请求等用户响应？**
> loop.py 不应该主动发 HTTP 请求，它只负责产生事件流。而且 loop.py 不知道前端在哪里，这样会造成强耦合。用 asyncio.Event + 共享 ApprovalManager 是更干净的解耦方式。

**Q：如果用户同时触发了两个高风险操作怎么办？**
> 每个审批请求有独立的 approval_id 和独立的 asyncio.Event 对象。resolve() 通过 approval_id 精确定位，只唤醒对应的那个协程，互不干扰。

**Q：超时时间 60 秒合理吗？**
> 这是可配置的（AgentLoop 构造函数的 approval_timeout 参数）。60 秒是经验值，给用户足够时间看清楚操作内容再决定。超时默认拒绝是安全默认值——不确定时宁可不做。

**Q：这个功能在生产环境还需要什么改进？**
> 至少三点：1）多用户场景下 ApprovalManager 需要按 session_id 隔离；2）审批记录应该写入审计日志，方便事后追溯；3）可以加"记住我的选择"功能，对同一个命令模式不重复询问。

## 第 1 类：基础概念

### Q1: 什么是 AI Agent？和普通聊天机器人有什么本质区别？

**面试级答案：**

> AI Agent 是一个**以大模型为决策核心**、**能自主使用工具完成多步任务**的系统。三个核心特征：
>
> **第一，目标驱动。** 用户给一个目标，Agent 自己规划需要哪些步骤。
>
> **第二，工具使用。** 它通过 Tool Calling 协议调用外部能力，比如读文件、执行代码。但工具本身不是大模型执行的——大模型只输出"我想调用 XX 工具"的意图，真正执行的是业务代码。
>
> **第三，循环与观察。** 这是 Agent 和普通聊天机器人最本质的区别。普通机器人是**单次问答**。Agent 是 **ReAct 循环**：调用工具 → 观察结果 → 继续推理 → 决定下一步动作，直到任务完成或达到终止条件。
>
> 举例：用户说"帮我统计 data.csv 有多少行"。普通机器人只能回答"请用 wc -l"。Agent 会主动执行：先调 read_file 打开文件 → 观察内容 → 调 exec 执行统计 → 把结果告诉用户。

**易错点：**
- "普通机器人只会想和说"——不准确，机器人不会"想"
- 漏说"多轮循环"

**追问 1：Agent 为什么需要循环？一次性调用不行吗？**
> 复杂任务没法一步到位。比如"分析 CSV 并画图"——大模型第一次调用时还不知道文件里有什么字段，必须先读了才能决定画什么图。循环是为了让 Agent 能**根据中间结果动态调整**后续动作。

**追问 2：Agent 的"自主性"体现在哪？**
> 体现在**是否继续循环、下一步用什么工具、传什么参数**这三个决策都是大模型做的，不是代码预先写死的。

**追问 3：你做过的 TinyAgent 项目里，ReAct 循环在哪个文件实现？**
> `core/loop.py` 的 `AgentLoop.run()`。核心是一个 `while` 循环，最多跑 10 轮（max_iterations）。每轮调用大模型，看返回有没有 tool_calls，有就执行工具继续下一轮，没有就 break。

---

### Q2: ReAct 模式是什么？怎么实现的？

**面试级答案：**

> ReAct 是 2022 年提出的 Agent 设计模式，全称 **Reasoning + Acting**，核心思想是让大模型在**推理和行动之间交替进行**。
>
> 三步循环：
>
> 1. **Reasoning（推理）**：大模型根据上下文思考"我下一步应该做什么"
> 2. **Acting（行动）**：大模型输出工具调用，由代码执行
> 3. **Observation（观察）**：工具执行结果被注入回上下文，大模型看到结果后继续推理
>
> 循环直到大模型认为任务完成，不再输出工具调用为止。
>
> 在 TinyAgent 里实现在 `core/loop.py` 的 `AgentLoop.run()` 方法。它是一个 while 循环，每轮调用一次大模型 API。如果包含 tool_calls，就执行工具、把结果追加到 messages，进入下一轮；如果没有，break 退出。同时设了 max_iterations=10 防死循环。
>
> 相比一次性输出所有步骤，ReAct 的优势是**每一步都能根据实际结果调整**。

**易错点：**
- 把 ReAct 当成 React 前端框架
- 漏掉"观察（Observation）"环节
- 只说流程不说设计思想

**追问 1：ReAct 和 CoT 有什么区别？**
> CoT 是纯推理链，模型只"想"不"做"。ReAct 在推理基础上加了行动和观察，能获取真实信息而不是靠模型自己编造。

**追问 2：ReAct 有什么缺点？**
> 一是每轮都要调用一次 API，延迟和成本会累积；二是模型可能陷入无效循环，所以需要 max_iterations 兜底。

**追问 3：除了 ReAct，还知道哪些 Agent 设计模式？**
> Plan-and-Execute（先规划再执行）、Reflexion（执行后自我反思）、Multi-Agent（多 Agent 协作）。

### Q3: Tool Calling 是怎么工作的？描述完整流程。

**面试级答案：**

> 5 个步骤：
>
> **第 1 步：注册工具定义。** 启动时 ToolRegistry 把每个工具转换成 OpenAI 兼容的 JSON Schema 格式（工具名、描述、参数结构）。每次 API 调用通过 tools 参数传给大模型。
>
> **第 2 步：组装上下文。** 用户发消息后，ContextBuilder 把 System Prompt + 历史对话 + 当前输入组装成 messages。
>
> **第 3 步：调用大模型。** AgentLoop 把 messages 和 tools 定义一起发给大模型。模型根据用户意图和工具描述**自主决定**是直接回答还是调用工具。如果调用工具，返回的不是执行结果，而是**意图声明**：`{name: "read_file", arguments: '{"path": "test.txt"}'}`。
>
> **第 4 步：代码执行工具。** ToolRegistry 解析 JSON 参数，找到工具类，调用 execute() 方法。
>
> **第 5 步：结果回传。** 工具结果以 `role: "tool"` 消息追加到 messages，再次调用大模型。模型看到结果后决定继续调用其他工具还是给出最终回答。
>
> 整个过程中，**大模型从头到尾没有执行过任何代码**，它只负责"说我想做什么"，真正做事的是 Python 代码。

**易错点：**
- 主语搞错（说"模型在阅读历史"——其实是 context.py 在做）
- 漏掉"大模型不执行工具"这个核心点
- 没提 JSON Schema / tools 参数

**追问 1：如果大模型返回的工具参数 JSON 格式不对怎么办？**
> ToolRegistry.execute() 里有 try/except json.JSONDecodeError，返回错误信息。这个错误会作为 tool 结果传回大模型，模型看到后通常会修正参数重试。

**追问 2：大模型怎么知道有哪些工具可用？**
> 通过 API 请求的 tools 参数。每个工具被转换成 JSON Schema 格式。大模型根据 description 判断什么时候用哪个工具。

**追问 3：如果大模型调用了不存在的工具名怎么办？**
> ToolRegistry.execute() 检查 `if name not in self.tools`，返回错误提示。错误同样回传给大模型。

### Q4: Agent 的"自主性"体现在哪里？是模型还是代码在做决策？

**面试级答案：**

> Agent 的自主性体现在：**大模型做决策，代码做执行。**
>
> 大模型自主决定四件事：
> 1. **要不要用工具**——简单问题直接回答，复杂任务调用工具
> 2. **用哪个工具**——从可用工具列表中选择最合适的
> 3. **传什么参数**——比如读哪个文件、执行什么命令
> 4. **什么时候结束**——判断任务是否完成
>
> 代码的角色是**提供能力和设定边界**：
> - 提供能力：注册工具、执行工具、管理上下文
> - 设定边界：max_iterations 防死循环、Shell 命令黑名单、文件大小截断
>
> Agent 的自主性是**有边界的自主性**。大模型在代码划定的安全范围内自由决策，但不能突破限制。
>
> 在 TinyAgent 里：loop.py 提供循环骨架，tools.py 提供工具和安全拦截，但循环里每一步走哪条路，完全由大模型返回值决定——代码里没有任何 `if user_message contains "读文件" then call read_file` 这样的硬编码逻辑。

**易错点：**
- 只说一个决策维度（"调不调工具"）
- 没说"代码的角色"
- 没提"有边界的自主性"

**追问 1：如果大模型做了错误的决策怎么办？**
> 短期靠 System Prompt 引导；长期靠 max_iterations 兜底。工具本身有安全检查，比如 Shell 工具拦截 rm -rf。

**追问 2：Agent 自主性越高越好吗？**
> 不是。自主性越高，不可控性越强。生产环境通常需要在关键节点加入人工确认（Human-in-the-loop），比如执行删除操作前让用户确认。

## 第 2 类：架构设计

### Q1: 介绍项目整体架构

**面试级答案：**

> 这个项目采用**模块化组合架构**，6 个核心文件分为 4 层：
>
> **第一层：HTTP 入口（app.py）**
> 基于 FastAPI，接收用户请求，通过 SSE 流式推送响应。不包含业务逻辑，只做"接收"和"转发"。
>
> **第二层：总控（agent.py）**
> TinyAgent 类是对外的唯一接口，内部组合了 5 个独立模块。负责协调整个对话流程。
>
> **第三层：核心引擎**
> - context.py：上下文构建器，把 System Prompt + 历史 + 当前输入组装成 messages
> - loop.py：ReAct 循环引擎，反复调用大模型直到任务完成
> - tools.py：工具注册中心，管理工具定义和执行
>
> **第四层：数据支撑**
> - memory.py：对话历史的存取和安全截断
> - skills.py：技能插件的加载和分级注入
>
> **协作流程：**
> 用户消息进来 → ContextBuilder 从 MemoryStore 取历史、从 SkillsLoader 取技能，组装 messages → AgentLoop 拿着 messages 和 ToolRegistry 的工具定义调用大模型 → 大模型返回工具调用时，ToolRegistry 执行并把结果注入 messages 继续循环 → 循环结束后新消息存回 MemoryStore → 全程通过 AsyncGenerator + SSE 流式输出给前端。
>
> **设计理念：** 每个模块职责单一、互不依赖，可以独立替换。比如想换记忆方案，只需要改 memory.py。

**易错点：**
- 答成流水账，没有清晰分层
- "agent 组装模型"——表述不准确（是组装上下文，不是组装模型）
- "通过 turn 函数送到用户界面"——内部事件不要和对外接口混淆

**追问 1：为什么用 AsyncGenerator 而不是 WebSocket？**
> SSE 是单向推送，实现简单，适合"服务器向客户端推流"。WebSocket 是双向通信，更复杂，这个场景用不到。

**追问 2：这个架构有什么局限性？**
> 主要是单实例、单用户。agent 是全局变量，多用户会共享同一份 memory。生产环境需要按 session 隔离。

### Q2: 用了什么设计模式？为什么选择组合而不是继承？

**面试级答案：**

> TinyAgent 在代码层面主要用了两个设计模式：
>
> **第一，组合模式（Composition）。** TinyAgent 类不继承任何基类，而是在内部组合了 5 个独立模块（MemoryStore、SkillsLoader、ToolRegistry、ContextBuilder、AgentLoop）。每个模块职责单一，通过构造函数注入。
>
> **为什么选组合而不是继承？** 三个原因：
> 1. **灵活性**：想换记忆方案（文件 → Redis），只需替换 MemoryStore，其他模块不动
> 2. **可测试性**：可以单独测试每个模块，不需要构造整个继承链
> 3. **避免继承层级膨胀**：如果用继承，"有工具的 Agent"、"有记忆的 Agent"、"有技能的 Agent" 会产生大量子类组合
>
> **第二，策略模式（Strategy）。** BaseTool 定义统一接口，每个具体工具（ReadFileTool、ShellTool）实现自己的 execute 逻辑。ToolRegistry 不关心具体是哪个工具，只要它实现了 execute 方法就能调用。新增工具只需要写一个类并注册，不用改任何已有代码。
>
> 一句话总结：**组合让模块可插拔，策略让工具可扩展。**

**核心概念：组合 vs 继承**

继承（"是一个"）：
```python
class MemoryAgent(ToolAgent):  # 层级深难维护
    pass
```

组合（"有一个"）：
```python
class TinyAgent:
    def __init__(self):
        self.memory = MemoryStore()    # Agent "有一个" 记忆
        self.tools = ToolRegistry()    # Agent "有一个" 工具
```

**追问 1：要新增一个工具，需要改哪些文件？**
> 只需要改 tools.py：写一个继承 BaseTool 的新类，在 ToolRegistry.\_\_init\_\_ 里 register 即可。其他文件不用动。

**追问 2：组合模式有什么缺点？**
> 模块之间的协调逻辑集中在 TinyAgent 里，如果模块间交互变复杂，TinyAgent 会变成"上帝类"。

### Q3: Agent 有哪些安全防护机制？大模型"失控"了怎么办？

**面试级答案：**

> TinyAgent 有 **4 层安全防护**：
>
> **第 1 层：循环保险丝（loop.py）**
> max_iterations = 10，防止大模型陷入死循环反复调用工具。无论什么情况，10 轮后强制停止。
>
> **第 2 层：Shell 命令黑名单（tools.py）**
> ShellTool 用正则拦截高危命令：rm -rf、shutdown/reboot、dd if=、mkfs/format、fork bomb。即使大模型"想"执行，代码层面直接拒绝。
>
> **第 3 层：命令执行超时**
> ShellTool 设置 timeout = 60 秒。超时强制 kill 进程，防止死循环脚本卡死服务器。
>
> **第 4 层：输出截断**
> read_file 文件超过 10000 字符截断；exec 命令输出超过 10000 字符截断。防止大模型读巨大文件撑爆上下文窗口。
>
> **总结：四层纵深防御**——循环层防死循环、命令层防危险操作、时间层防卡死、数据层防溢出。

**易错点：**
- 只答 max_iterations 一个点（这是最常见的失分点）

**追问 1：黑名单能完全防御吗？比如 `python -c "import os; os.system('rm -rf /')"`？**
> 这是当前实现的局限。正则黑名单只能拦截已知模式。生产环境应该用**沙箱**（Docker 容器、gVisor）隔离执行环境，即使命令执行了也不影响宿主机。

**追问 2：max_iterations 设成 10 合理吗？**
> 太小（3）：复杂任务做不完。太大（100）：死循环会浪费大量 API 费用。10 是经验值，多数任务 2-3 轮完成。

**追问 3：生产环境还需要什么安全措施？**
> 用户认证、请求频率限制、沙箱执行、敏感信息过滤、审计日志。

### Q4: 怎么把单用户改成支持多用户并发？

**面试级答案：**

> 当前架构核心问题：app.py 第 28 行的 `agent = TinyAgent(...)` 是**全局单例**，所有用户共享同一实例。会导致：
> 1. 记忆混乱（用户 A 看到 B 的对话）
> 2. 历史文件冲突（并发写入丢数据）
> 3. token 统计不准
>
> 改造方案 4 步：
>
> **第 1 步：引入 session_id**
> 前端每个用户生成 UUID，通过 Cookie 或请求头带上来。
>
> **第 2 步：Agent 实例池化**
> ```python
> agents = {}
> def get_agent(session_id):
>     if session_id not in agents:
>         agents[session_id] = TinyAgent(...)
>     return agents[session_id]
> ```
>
> **第 3 步：记忆按 session 隔离**
> MemoryStore 已经支持 session_id 参数（memory.py 第 12 行），改造后每个 session 写入独立文件。
>
> **第 4 步：解决并发写入**
> ```python
> import asyncio
> self.lock = asyncio.Lock()
>
> async def add_message(self, msg):
>     async with self.lock:
>         self.messages.append(msg)
>         self._save_history()
> ```
>
> **进一步工程化：**
> - 内存池随用户增多无限增长 → LRU 缓存淘汰
> - 文件存储扛不住高并发 → Redis 或数据库
> - 单机扛不住更多用户 → agent 状态外部化，多节点共享
> - 用户认证（JWT）防 session_id 冒用

**追问 1：为什么不直接每次请求都新建 Agent？**
> MemoryStore 启动时要从文件加载历史，每次新建会重复加载性能差。而且并发请求新建同一用户的 Agent，又回到并发写入问题。

**追问 2：用 Redis 替代文件存储有什么好处？**
> 高并发性能好（内存操作）、原子操作支持（INCR、LPUSH 无锁更新）、多机共享（多节点连同一个 Redis）。
